# レビューレポートテンプレート

## ヘッダー

```markdown
# [分析タイプ] 分析レポート

**日付**: YYYY-MM-DD HH:MM
**レビュアー**: Claude (review-worker)
**対象**: #{issue-number} {issue-title}
**ロール**: [plan|requirements|design|research]
```

## セクション

### 1. サマリー

概要（2-3文）:
- レビュー対象
- 総合評価（Good/Needs Work/Critical）
- 主要な発見事項

### 問題サマリー

| 深刻度 | 件数 |
|--------|------|
| Critical | {n} |
| High | {n} |
| Medium | {n} |
| Low | {n} |
| **合計** | **{n}** |

問題が 0 件の場合はこのテーブルを省略し、「問題は検出されませんでした」と記載する。

### 2. 重大な問題

**修正必須**（深刻度定義参照）

| 問題 | 場所 | 深刻度 | 説明 |
|------|------|--------|------|
| 問題名 | セクション/箇所 | Critical/High | 問題の内容と理由 |

### 3. 改善提案

**推奨される変更**（深刻度定義参照）

| 問題 | 場所 | 深刻度 | 説明 |
|------|------|--------|------|
| 問題名 | セクション/箇所 | Medium/Low | 改善内容 |

### 4. 良い実践

**ポジティブな発見**

- 良くできている点
- 正しく適用されたパターン
- 他で再利用すべき良い例

### 5. 推奨事項

**優先度順のアクション項目**

1. **即時対応**: Critical な修正
2. **次フローへ進む前に**: 重要な改善
3. **将来**: 技術的負債

### 6. 成果物レビュー結果（成果物レビュー対象が存在する場合のみ）

**PR 成果物として作成・更新された GitHub ドキュメント（Discussion / Issue）のレビュー結果**

> このセクションは「成果物レビュー対象:」が指定されている場合のみ表示する。成果物が存在しない場合は省略する。

| 成果物 | 種別 | 評価 | 指摘内容 |
|--------|------|------|----------|
| #N タイトル | Discussion/Issue | OK/要修正 | 指摘の詳細（無ければ「—」） |

**指摘詳細**（指摘がある場合のみ）:

- `#N`: {指摘内容の詳細記述}

### 7. 設計要否（requirements ロールの場合のみ）

> このセクションは requirements ロールの場合のみ表示する。他のロールでは省略する。

**設計要否:** NEEDED / NOT_NEEDED

判定根拠: {NEEDED / NOT_NEEDED と判定した主要な根拠を 1-2 文で記載}

## レポート例

```markdown
# 計画レビューレポート

**日付**: 2026-05-20 14:30
**レビュアー**: Claude (review-worker)
**対象**: #1234 ユーザー認証フローの再設計
**ロール**: plan

## サマリー

計画 Issue #1234 をレビュー。全体評価は **Needs Work**。タスク粒度は適切だが、要件 Issue #1200 の AC-3（パスワードリセット時のレート制限）に対応するタスクが計画から欠落している。

### 問題サマリー

| 深刻度 | 件数 |
|--------|------|
| Critical | 1 |
| High | 1 |
| Medium | 1 |
| Low | 0 |
| **合計** | **3** |

## 重大な問題

| 問題 | 場所 | 深刻度 | 説明 |
|------|------|--------|------|
| AC-3 カバレッジ欠落 | タスクリスト | Critical | パスワードリセットのレート制限タスクが未定義 |
| テスト戦略未記載 | テスト計画 | High | Server Actions の単体テスト方針が不明 |

## 改善提案

| 問題 | 場所 | 深刻度 | 説明 |
|------|------|--------|------|
| 依存タスクの順序 | タスクリスト | Medium | DB マイグレーションを最初のタスクに移動推奨 |

## 良い実践

- 要件 Issue へのリンクが明確
- リスク評価セクションが充実
- 各タスクに想定工数が記載されている

## 推奨事項

1. **即時対応**: AC-3 対応タスクを追加
2. **次フローへ進む前に**: テスト戦略セクションを記載
3. **将来**: タスク順序の最適化
```

## 深刻度定義

| レベル | 影響 | アクション |
|--------|------|----------|
| **Critical** | 要件欠落、重大な曖昧さ、実装不可能な要件 | 次フロー進行前に修正 |
| **High** | 重要な情報の不足、整合性の問題 | 修正を強く推奨 |
| **Medium** | 改善余地、明確化の余地 | 修正推奨 |
| **Low** | スタイルの問題、軽微な改善 | 後日対応可 |
